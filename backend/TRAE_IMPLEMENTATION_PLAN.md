# Backend 项目实施方案

## 1. 项目目标

后端目标是搭建一套可演示、可追溯、可扩展的竞品分析智能体系统，完成以下闭环：

1. 接收用户任务，抽取行业、竞品对象和分析重点。
2. 由多个角色化智能体分阶段完成规划、采集、分析、撰写与质检。
3. 将分析过程中的结构化产物、证据、事件、报告与修订记录持久化。
4. 对外提供可被前端消费的运行态接口、工作区接口、报告追问接口与实时事件流接口。
5. 支持质检打回、局部返工、多轮追问、报告修订和结果复盘。

## 2. 总体建设思路

后端不采用“单模型一次性生成报告”的路线，而采用“任务编排 + 多智能体协作 + 结构化状态驱动”的方式建设。整个实施分为五个阶段：

1. 先搭建状态模型和持久化骨架。
2. 再搭建工作流编排和角色化 Agent。
3. 再补采集、分析、报告生成与 QA 闭环。
4. 再补实时事件、报告追问和人工干预接口。
5. 最后补齐测试、可观测性和异常恢复机制。

在实施过程中，编码助手主要承担代码检索、接口草拟、测试补齐、重构建议与回归检查；开发者负责目标拆解、边界约束、方案取舍与最终验收。所有关键能力都采用多轮迭代推进，而不是一次性生成后直接交付。

## 3. 分阶段实施计划

### 阶段一：基础骨架

本阶段先不追求“智能”，重点是把系统最难改的底层骨架稳定下来。

建设内容：

- 设计统一的 `RunState`、`Report`、`Finding`、`Evidence`、`QAOutput` 等核心数据模型。
- 设计 SQLite 持久化层，保证状态、事件、报告修订、对话 turn 都可落盘。
- 设计基础 API：
  - 创建任务
  - 查询运行状态
  - 查询工作区快照
  - 查询事件列表
- 明确事件模型，保证后续所有 Agent 行为都有统一记录格式。

这一阶段，编码助手的主要作用是快速比较多种模型定义方式、辅助补齐序列化与反序列化代码、提前生成最小测试样例。开发者重点审核模型边界是否稳定，避免后续频繁推翻数据库结构。

### 阶段二：多智能体工作流

本阶段开始搭建真正的多 Agent 协作链路。

建设内容：

- 设计 Manager / Orchestrator / Collector / Analyst / Writer / QA 等角色。
- 为每个 Agent 定义清晰输入、输出与结构化 handoff。
- 引入统一编排框架，保证任务不是自由对话，而是受控流转。
- 设计阶段间可回溯的事件、trace 与节点状态。

实施方式：

- 先由开发者定义每个 Agent 的职责边界和禁止越权事项。
- 再由编码助手辅助生成各 Agent 的基础类、Prompt 框架、Schema 校验代码。
- 第一轮只打通主路径：规划 -> 采集 -> 分析 -> 撰写 -> 质检。
- 第二轮再补返工路径和失败恢复路径。

### 阶段三：证据链与质量闭环

这个阶段的核心不是多加几个 Agent，而是让输出真正可用、可追溯。

建设内容：

- 采集层支持多来源检索和规范化证据抽取。
- 分析层严格产出结构化字段，而不是自由文本总结。
- 报告层把分析结果组织为固定章节和统一格式。
- QA 层不仅给出通过/不通过结论，还能指出返工对象和返工内容。
- 每条核心结论都绑定可定位的证据来源。

实施方式：

- 先建立最小闭环：QA 能把明显缺失项打回 Collector 或 Analyst。
- 再增加局部 rework 机制，避免每次整条链路全量重跑。
- 编码助手负责快速补测试样本和异常 case，开发者重点判断“什么算可打回、什么算可以通过”。

### 阶段四：对外接口与实时交互

在主流程稳定后，再补面向前端演示和交互的能力。

建设内容：

- 提供工作区快照接口，供前端展示各阶段产物。
- 提供事件流接口，支持运行过程中的实时可视化。
- 提供报告 Markdown 下载、修订、问卷设计等操作接口。
- 提供报告追问能力，支持对既有报告做多轮问答。
- 在报告追问中支持 SSE 流式返回，做到逐步输出。

实施方式：

- 先做异步 turn 模型，保证追问结果可持久化。
- 再做 SSE 事件流，先推阶段进度，再推正文增量。
- 对“正文流”和“状态流”分别建模，避免前端混用。
- 编码助手协助设计事件 schema、补 EventSource 接口与 smoke 脚本，开发者重点判断交互节奏与恢复策略。

### 阶段五：稳定性、测试与演示准备

本阶段重点是把系统从“能跑”提升到“可演示、可复盘、可解释”。

建设内容：

- 完善网络重试、超时、降级和异常日志。
- 补齐单元测试、API 测试、live smoke 测试和关键脚本。
- 为关键节点补 trace、prompt、输入输出和 token 记录。
- 输出架构说明、接口说明、角色协议和实施记录。

这一阶段，编码助手特别适合承担：

- 自动生成和扩展测试脚本
- 比对失败日志并定位改动影响面
- 草拟文档和实施说明

开发者重点关注：哪些测试有真实价值，哪些只是表面覆盖；哪些文档值得给外部看，哪些只保留内部使用。

## 4. 协作方式

项目开发不采用“需求一句话 -> 一次性生成全部代码”的方式，而采用以下协作节奏：

1. 先由开发者明确目标、边界和验收口径。
2. 编码助手先做代码结构分析、调用链梳理和改造清单输出。
3. 双方确认后，编码助手按模块提交增量补丁。
4. 每轮补丁都配套最小验证。
5. 开发者根据真实运行结果决定是否收敛、回退或重构。

这种方式更适合复杂工程，因为它允许：

- 在多文件改造前先看影响面
- 在关键路径上线前先用 mock 和 smoke 验证
- 在方案不理想时快速调整，而不是被一次性生成的代码绑死

## 5. 关键里程碑

### M1：状态与事件骨架可运行

验收标准：

- 能创建 run
- 能保存 state
- 能查询事件和工作区基础信息

### M2：多 Agent 主流程打通

验收标准：

- 至少五个核心阶段可顺序执行
- 每阶段输入输出结构清晰
- 报告可成功生成

### M3：QA 闭环可触发

验收标准：

- QA 能识别问题
- 能指定返工对象
- 返工后结果有变化

### M4：报告追问与流式输出可演示

验收标准：

- 能对已有报告发起追问
- 能看到实时进度与正文增量输出
- 最终结果可持久化并回查

### M5：测试与文档齐备

验收标准：

- 关键链路有 smoke test
- 核心模块有单元或 API 级测试
- 文档能支撑外部快速理解系统结构与实施过程

## 6. 风险与应对

### 风险一：智能体边界不清，最后退化成单体脚本

应对方式：

- 每个 Agent 强制结构化输入输出
- Manager 不直接代替子 Agent 产出业务结果
- 所有 handoff 和返工记录都要落盘

### 风险二：多轮追问接口复杂，前后端协议频繁变动

应对方式：

- 先冻结 turn 模型
- 再逐步迭代事件 schema
- 保留非流式结果查询接口作为兜底

### 风险三：测试难写，容易只验证 happy path

应对方式：

- 除主路径外，优先补 fallback、rework、协议错误、断流恢复等测试
- 用 mock LLM 和 smoke 脚本分层验证

## 7. 最终交付

后端最终交付内容包括：

- 多 Agent 编排服务
- 结构化状态与事件持久化
- 工作区与事件流 API
- 报告追问与流式接口
- 返工与 QA 闭环能力
- 可复现的测试脚本和实施文档

## 8. 结语

后端建设的关键不是让模型“多说一点”，而是把任务拆解、证据组织、状态推进、返工闭环、实时事件和持久化能力逐步落到工程系统里。  
整个项目适合采用“开发者定目标、编码助手做增量协作、每轮都有验证”的实施方式，这样既能保持迭代速度，也能保证复杂系统的稳定性与可解释性。
