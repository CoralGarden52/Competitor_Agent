# API 接口文档

## 1. 文档概述

本项目围绕“竞品分析任务”提供了一组完整的后端接口，支持从任务创建、运行过程查看，到报告修订、问卷生成与问卷导出的整条使用链路。  
这些接口对应的不是零散功能点，而是一套完整的竞品分析工作流：用户提交研究任务后，系统会自动完成规划、采集、分析、质检和报告生成，随后还支持继续追问和问卷验证。

整体接口设计强调三点：

- 入口统一，便于前端工作台和后续系统集成调用；
- 返回结构清晰，便于展示运行状态、分析结果和证据来源；
- 支持流式交互，便于实时展示多 Agent 协作过程。

## 2. 接口体系说明

从使用方式看，系统接口主要分为五组：

1. 运行管理接口  
用于创建竞品分析任务、查询运行状态、获取事件流和工作区快照。

2. 报告交互接口  
用于获取、下载、修订报告，并围绕既有报告继续发起追问。

3. 问卷接口  
用于根据竞品分析报告生成问卷，并导出到问卷星。

4. 采集预览接口  
用于在正式运行前预览行业识别、竞品发现和分析字段规划结果。

5. Schema 与配置接口  
用于查看当前可用的分析 Schema 注册表和运行时配置。

从整体上看，这五组接口分别对应了系统的不同职责：

- 运行管理接口负责让任务“跑起来”；
- 报告交互接口负责让结果“用起来”；
- 问卷接口负责让分析结论“继续验证下去”；
- 采集预览接口负责让任务在正式执行前“先看清方向”；
- Schema 与配置接口负责让系统的分析结构和运行环境“可查看、可解释”。

这样划分的好处是，读者不需要先了解内部代码结构，也能直接从接口分组中理解产品能力边界。

## 2.1 接口设计与系统主链路的关系

本项目的接口并不是围绕“若干孤立功能按钮”设计的，而是围绕一条完整的竞品分析主链路设计的。  
从调用顺序看，一次典型任务通常会经历以下阶段：

1. 提交分析任务，系统创建运行实例；
2. 系统自动进入规划、采集、分析、质检和写作流程；
3. 前端通过工作区接口与流式接口持续获取阶段状态和过程信息；
4. 任务结束后，用户可继续围绕报告追问、修订和补充；
5. 在需要验证结论时，再进一步生成问卷并导出。

因此，接口设计体现的不是“后端提供了什么能力”，而是“竞品分析工作流如何在产品中真正跑起来”。

## 2.2 统一返回与状态组织方式

为了便于前端工作台展示和后续系统集成，系统在接口层面尽量保持统一的状态组织方式。

- 运行相关接口以 `run_id` 作为主索引，围绕一次完整任务聚合过程数据与结果数据。
- 报告、问卷、追问等后续能力并不是独立系统，而是继续依附于既有 `run_id` 展开，保证同一任务下的信息连续性。
- 工作区快照接口负责提供聚合视图，适合页面初始化与刷新恢复；事件流接口负责提供增量更新，适合实时展示；明细接口负责补充某一类数据的细节读取。

这种设计方式可以让评审者更直观地看到：系统不是简单地“生成一份报告”，而是在一次任务内持续沉淀状态、证据、分析结果和后续互动内容。

## 4. 关键接口详细说明

### 4.1 创建竞品分析任务

**接口**

`POST /runs`

**作用**

创建一条新的竞品分析任务。任务创建成功后，系统会异步进入多阶段工作流，依次完成任务规划、证据采集、结构化分析、质量审查和报告生成。

从业务视角看，这个接口是整套系统的起点。  
用户只需要给出一个竞品研究目标，系统就会围绕这一目标自动展开后续工作，而不需要用户手工拆解成“先搜什么、再比什么、最后怎么写”。

**典型输入**

```json
{
  "prompt": "分析通用 AI 智能体平台的主要竞品，重点关注产品能力、定价和用户反馈",
  "industry_hint": "AI Agent",
  "competitor_hints": ["OpenAI", "Anthropic"],
  "focus_points": ["产品能力", "价格策略", "用户评价"]
}
```

**典型输出**

```json
{
  "run_id": "run_xxx",
  "status": "queued"
}
```

**说明**

- `prompt` 为必填项，用于描述分析任务。
- 其余字段用于补充行业、竞品线索和关注维度，帮助系统更快形成合理的分析规划。

**设计特点**

- 接口允许用户只输入开放式任务描述，也允许补充竞品线索和关注维度，兼顾探索式任务和目标明确的任务。
- 任务创建后立即返回 `run_id`，后续所有阶段状态、报告内容和问卷结果都围绕该标识继续展开，便于形成完整任务档案。

### 4.2 获取工作区快照

**接口**

`GET /runs/{run_id}/workspace`

**作用**

返回竞品分析工作台展示所需的聚合数据，是最核心的读取接口之一。  
它不仅返回任务状态，还会同时返回报告内容、阶段进度、事件摘要、分析字段、QA 结果和问卷信息。

对于前端工作台来说，这个接口几乎承担了“任务总览页”的全部职责。  
它的价值不只是方便读取数据，更重要的是把一次复杂任务中的多种信息聚合到同一个视图里，让用户能够从一个入口快速理解当前任务的整体状态。

**主要信息类型**

- 任务摘要与当前状态
- 阶段状态与阶段说明
- 动态 Schema 规划结果
- 工作区事件与可观测信息
- 报告内容与报告修订状态
- QA 检查结果与返工信息
- 问卷内容与导出结果

**适用场景**

- 打开历史任务
- 页面刷新后恢复完整状态
- 流式连接中断后的状态补全

**设计特点**

- 该接口把阶段状态、分析字段、QA 结果、报告内容和问卷信息聚合到同一视图中，适合作为工作台核心接口。
- 它反映的是“当前任务已经走到哪里、产出了什么、还缺什么”，因此也是理解多 Agent 工作流的最佳入口。

### 4.3 订阅运行期事件流

**接口**

`GET /runs/{run_id}/stream`

**作用**

通过 `SSE` 持续返回运行过程中的阶段更新、工作区更新和结束状态，使使用者能够实时看到系统如何从输入任务逐步推进到输出报告。

在产品体验上，这个接口非常关键。  
竞品分析并不是瞬时完成的短任务，如果只能在结束后一次性看到最终结果，用户往往很难判断系统当前是否正常运行、是否已经进入返工阶段、报告是否正在生成。  
流式接口的存在，使任务过程本身变得可见，也让多 Agent 协作不再只是后台黑盒。

**主要事件类型**

- `workspace`：工作区全量更新
- `run_event`：运行事件增量更新
- `task_summary`：任务摘要更新
- `run_done`：任务结束
- `error`：异常状态

**使用价值**

- 展示多 Agent 协作过程
- 展示 QA 打回与返工过程
- 支撑报告生成过程中的实时可视化体验

**设计特点**

- 流式接口并不只返回最终文本，而是返回阶段推进、状态变化和增量事件，因此能够真实反映系统执行过程。
- 对于竞品分析这类耗时较长的任务，实时展示过程比单纯等待结果更重要，它能明显提升系统的透明度和可解释性。

### 4.4 报告下载与修订

**接口**

- `GET /runs/{run_id}/report.md`
- `PATCH /runs/{run_id}/report`

**作用**

支持对已生成的报告进行下载、保存和继续修订。  
这使系统输出不再是一次性结果，而是可持续完善的研究产物。

在真实工作中，一份报告通常会经过多轮打磨。  
因此，这组接口的意义不只是“允许修改文本”，而是让报告成为一个可以继续维护的交付物，后续表达调整、补充说明和汇报定制都可以在既有成果上继续展开。

**设计特点**

- 报告实体在系统中被视为可持续维护的研究结果，而不是一次性生成后不可修改的终态文本。
- 这一设计让系统更贴近真实工作流，因为竞品分析报告通常需要经历多轮内部讨论、表达调整和重点补充。

### 4.5 报告追问与流式回答

**接口**

- `POST /runs/{run_id}/chat`
- `GET /runs/{run_id}/chat`
- `GET /runs/{run_id}/chat/{turn_id}`
- `GET /runs/{run_id}/chat/{turn_id}/stream`

**作用**

支持围绕已有报告继续提问，例如追问某一结论的依据、要求改写某一章节，或补充某个竞品信息。  
系统会基于已有报告、工作记忆和必要的补充检索继续回答，并支持流式输出。

这组接口体现的是系统从“生成结果”走向“持续协作”的能力。  
用户并不需要重新创建任务，就可以围绕现有成果继续提问和修订，这使系统更像一个长期协作助手，而不是一次性内容生成器。

**主要事件类型**

- `chat_bootstrap`
- `chat_snapshot`
- `chat_progress`
- `chat_done`
- `chat_error`
- `heartbeat`

**设计特点**

- 报告追问能力建立在既有报告和任务记忆之上，而不是重新发起一轮完整分析，因此能够显著降低二次修改成本。
- 当用户追问事实细节或要求补充证据时，系统还可以继续结合工具检索进行回答，使报告具备持续延展能力。

### 4.6 采集预览

**接口**

`POST /collector/preview`

**作用**

在正式执行完整分析链路之前，先根据任务描述返回行业识别结果、候选竞品和分析字段规划，帮助使用者提前判断系统将如何理解并执行该任务。

这个接口的意义在于“先校准，再执行”。  
对于开放式任务，如果一开始竞品范围或分析维度就偏了，后面整条链路都会跟着偏。  
因此，采集预览接口相当于给用户提供了一个低成本的确认步骤，让任务在正式运行前就能先校验方向是否合理。

**典型价值**

- 提前验证竞品范围是否合理
- 提前查看分析维度是否覆盖目标问题
- 演示系统的自动行业识别和动态 Schema 规划能力

**设计特点**

- 采集预览接口相当于主流程的“前置检查点”，能够帮助使用者在正式执行前先确认系统对任务的理解是否合理。
- 对于开放式需求，这一步尤其重要，因为它可以尽早发现竞品范围过宽、字段偏离问题等潜在风险。

### 4.7 问卷生成与导出

**接口**

- `POST /runs/{run_id}/questionnaire`
- `PATCH /runs/{run_id}/questionnaire`
- `POST /runs/{run_id}/questionnaire/export/wenjuan`

**作用**

系统能够将竞品分析报告进一步转换为用户调研问卷，并在确认后导出为问卷星链接，形成“分析结论 -> 用户验证”的延伸链路。

这组接口所体现的，并不是一个单独附加的小功能，而是系统设计理念的一部分。  
项目并不把竞品分析视为“报告写完就结束”，而是希望让分析结果能够继续用于后续验证，因此才会把问卷能力直接纳入同一条任务链路之中。

**导出结果通常包含**

- `provider`
- `status`
- `title`
- `url`
- `vid`
- `exported_at`

**设计特点**

- 问卷接口并不是独立问卷系统，而是建立在既有报告结果之上的后续能力，因此能够把“公开信息研究”自然延伸到“用户验证”。
- 这种接口组织方式体现了本项目的一条重要设计思路：竞品分析并不止步于形成结论，而是要为后续验证和决策服务。

## 5. 推荐使用路径

若从完整业务链路理解本系统接口能力，建议按以下顺序使用：

1. 调用 `POST /runs` 创建分析任务
2. 通过 `GET /runs/{run_id}/workspace` 查看工作区快照
3. 通过 `GET /runs/{run_id}/stream` 观察实时阶段推进
4. 在报告生成后，使用 `POST /runs/{run_id}/chat` 继续追问
5. 在报告确认后，使用 `POST /runs/{run_id}/questionnaire` 生成问卷
6. 最后使用 `POST /runs/{run_id}/questionnaire/export/wenjuan` 导出问卷链接

## 6. 设计特点总结

本项目接口设计具有以下特点：

- 围绕竞品分析任务完整链路设计，而不是零散工具接口集合
- 支持结构化工作区快照，便于前端统一展示
- 支持 `SSE` 流式事件，能够真实反映系统运行过程
- 支持报告后的持续修订和问卷扩展，增强结果复用价值
- 支持采集预览、Schema 查看和质量闭环，体现系统工程化能力

从接口层可以看出，本项目更像一套可持续运行的竞品研究工作台，而不是单次调用后端生成文本的简单服务。  
接口设计之所以较为完整，是因为系统需要承载的不仅是“生成结果”，更是“展示过程、支持修订、延伸验证和沉淀状态”。
