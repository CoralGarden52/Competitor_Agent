# 评测方案与样例

## 1. 文档概述

本项目的评测重点不只是“最终报告是否写得通顺”，而是整套竞品分析系统是否真正完成了任务。  
因此，评测时不仅要看最终文本，也要看它是否正确理解了任务、是否找到了合理的竞品、是否保留了证据来源、是否具备返工修正能力，以及结果是否还能继续被使用。

由于本项目属于多 Agent 协作系统，单纯比较一段最终文本很难完整反映它的能力，因此更适合采用“结构化指标 + 过程回放 + 人工质检”相结合的评估方式。

## 2. 评测目标

本项目评测主要围绕以下六类能力展开：

1. 能否正确理解用户任务并识别所属行业；
2. 能否自动发现合理的直接竞品与替代竞品；
3. 能否生成结构一致、可横向比较的分析 Schema；
4. 能否采集到来源明确、字段归属清晰的结构化证据；
5. 能否输出覆盖较完整、结论有依据的竞品分析报告；
6. 能否在 QA 发现问题后触发有效返工并改进结果。

## 3. 评测维度

## 3.1 任务理解与分析规划

这一维度重点评估系统是否能够把开放式任务真正转化为可执行分析计划。  
竞品分析任务在真实场景中往往不会被用户描述得非常标准，很多时候只有一句目标、一两个线索，甚至只是一个模糊的问题方向。  
因此，这一维度实际上考察的是系统有没有能力把自然语言需求拆开，判断所属行业、识别可能的竞品范围，并进一步规划出后续分析要围绕哪些维度展开。

**关注点**

- 行业识别是否合理
- 候选竞品是否覆盖主要直接竞品与替代竞品
- 动态 Schema 是否覆盖任务关心的核心问题
- 用户补充的竞品线索和关注点是否被吸收进规划结果

## 3.2 采集与证据质量

这一维度重点评估系统对公开信息的检索、筛选和组织能力。  
对于竞品分析来说，决定结果质量的核心并不只是模型写得好不好，而是前面拿到的证据是否足够、是否相关、是否来源明确。  
如果证据本身杂乱、单一或无法追溯，那么后续再好的生成能力也很难真正提高结果可信度。

**关注点**

- 证据来源是否多样
- 证据是否保留 URL 与来源信息
- 是否存在大量重复来源或明显无关内容
- 价格、功能、用户反馈等关键字段是否具备足够支撑
- 在证据不足时是否能触发更深入的补采和交叉核验

## 3.3 分析与报告质量

这一维度重点评估系统对结构化证据的归纳能力和最终报告质量。  
这里关心的不是报告有没有写得很长，而是它能否围绕关键信息形成稳定、可比较、可用于讨论的结论。  
一份好的竞品分析报告应该能够回答“各家产品差异在哪里、核心判断依据是什么、哪些地方还存在信息不足”，而不是停留在泛泛介绍层面。

**关注点**

- 字段覆盖是否完整
- 关键结论是否与证据一致
- 报告结构是否清晰稳定
- 是否能够形成横向比较，而不仅是逐家介绍
- 是否能够支持后续汇报、讨论和复用

## 3.4 质量闭环与修正能力

这一维度最能体现本项目与普通一次性生成型方案的差异。  
很多系统虽然能生成报告，但一旦结果有问题，往往只能重新来一遍，或者只能由人工判断下一步该补哪里。  
本项目更强调的是：系统在发现问题之后，是否能够明确指出问题属于哪个竞品、哪个字段或哪个章节，并继续推动后续返工修正。

**关注点**

- QA 是否能够识别证据不足、字段缺失和报告缺口
- 返工是否能定位到具体竞品、字段或章节
- 返工后覆盖率和结果质量是否提升
- 是否能够避免无意义的全链路反复重跑

## 3.5 结果延展能力

这一维度用于评估系统是否能够在报告之后继续发挥价值。  
真实业务中的竞品分析通常不会在“报告生成完成”这一刻结束，后面往往还会继续追问、修改措辞、补充证据，甚至把结论转化为用户调研问题。  
因此，这一维度本质上是在判断：系统输出的是一份“一次性结果”，还是一套可以继续工作的研究成果。

**关注点**

- 是否支持围绕报告继续追问和修订
- 是否支持将报告进一步转化为调研问卷
- 是否能形成“分析结论 -> 用户验证”的延伸链路

## 4. 评估方法

## 4.1 人工质检评估

由使用者阅读系统输出，重点判断：

- 任务理解是否准确
- 竞品选择是否合理
- 证据是否可信
- 结论是否回应原始问题
- 报告是否可用于进一步汇报或决策参考

这种方式更适合判断系统在真实使用场景中的实际价值。  
因为很多问题无法只通过结构化指标看出来，例如结论是否真正回应了原始问题、报告是否适合拿去开会讨论、表达是否足够清晰等。  
人工质检的意义，就是从“结果是否可被人使用”的角度补足纯自动评估的不足。

## 4.2 结构化指标评估

结合系统输出的结构化中间结果，可统计如下指标：

- 竞品覆盖率
- 字段覆盖率
- 单字段证据条数
- 来源站点多样性
- QA 打回次数
- 返工后覆盖率提升情况
- 报告追问后的补充有效性

这种方式更适合观察系统在“过程质量”上的工程化特征。  
它能够帮助使用者看到系统不只是“生成了一份内容”，而是在运行过程中真正形成了覆盖度、来源质量、返工效果等可观测信号。  
对于多 Agent 系统来说，这类指标往往比单纯比较最终文本更能体现实际使用价值。

### 指标说明建议

为了让评测结果更具可解释性，建议为上述指标补充统一口径，例如：

- 竞品覆盖率：目标竞品中被成功纳入分析范围的比例；
- 字段覆盖率：计划分析字段中，已经形成有效分析结果的比例；
- 单字段证据条数：每个关键字段对应的有效证据数量；
- 来源站点多样性：同一字段下独立来源站点的数量；
- QA 打回次数：一次任务中被 QA 触发返工的轮次；
- 返工后覆盖率提升情况：返工前后字段覆盖率的变化；
- 报告追问后的补充有效性：追问后新增内容是否真正补足了原有缺口。

如果需要形成更完整的使用记录，还可以进一步补充人工修正率、报告完成时长和问卷导出完成率等指标。

## 4.3 样例回放评估

对固定样例任务保存完整运行结果，并比较：

- 首轮结果
- QA 打回后的返工结果
- 最终报告
- 后续追问与问卷结果

这种方式适合用于回顾完整任务过程，因为它可以同时反映系统的主链路和返工修正过程。  
相比单独查看最终报告，样例回放更能帮助使用者理解系统的真实工作过程：任务是如何被理解的，证据是如何被采集和组织的，哪里触发了 QA，返工后又发生了什么变化。

## 4.4 建议的综合评估方式

为了更全面地反映项目能力，建议将三类评估方式结合起来使用：

1. 先通过人工质检判断结论是否可信、报告是否可用；
2. 再通过结构化指标查看系统在覆盖率、来源质量和返工效果上的表现；
3. 最后用样例回放查看系统的真实运行过程和返工修正过程。

这种组合方式既能回答“结果好不好”，也能回答“结果是怎么来的、遇到问题如何修正”。

## 5. 评测样例

## 5.1 样例一：在线主流文档软件竞品分析（目标产品：飞书）

**输入示例**

```text
给出在线主流文档软件的竞品分析，目标产品是飞书。
```

**预期表现**

- 系统能够识别任务主题为在线文档/协作文档类软件
- 报告主体围绕飞书展开，而不是泛泛罗列办公协作工具
- 候选竞品能够覆盖主流在线文档或协作文档产品，并形成横向比较
- 报告中能够较稳定地覆盖文档能力、协同方式、定价模式、用户反馈等核心维度
- 最终输出体现“目标产品 vs 主要竞品”的对照关系，而不是逐家独立介绍

**为什么适合作为评测样例**

这一类任务能够同时检验系统的三个关键能力：  
第一，能否从“在线主流文档软件”这一表述中识别出正确的任务边界，而不是把范围扩展到整个办公协作软件集合；  
第二，能否理解“目标产品是飞书”意味着输出需要围绕飞书展开，而不是仅做行业概览；  
第三，能否在文档、协同、定价和用户反馈等维度上形成可用的横向结论。  
如果系统在这一样例中表现稳定，通常说明它已经能够较好地处理“有明确目标产品”的竞品分析任务。

## 5.2 样例二：在线会议软件竞品分析（指定需包含腾讯会议）

**输入示例**

```text
给出在线会议软件的竞品分析，并且由用户指定相关竞品需要包含腾讯会议。
```

**预期表现**

- 系统能够识别任务主题为在线会议软件，而不是泛化为整个协作办公或企业 SaaS 场景
- 在候选竞品规划中，腾讯会议必须被纳入分析范围
- 若系统还补充其他竞品，这些补充对象应与在线会议场景保持强相关
- 报告中能够较稳定地覆盖会议能力、稳定性/体验、企业适配、定价方式和用户反馈等维度
- 最终输出能够体现腾讯会议与其他主要会议产品之间的差异，而不是只把腾讯会议作为普通列表项

**为什么适合作为评测样例**

这一类任务特别适合检验系统对“用户显式约束”的处理能力。  
用户并不只是给出一个行业主题，还额外指出相关竞品中必须包含腾讯会议。  
这意味着系统不仅要完成行业识别和竞品扩展，还要保证最终规划结果满足用户约束。  
如果系统在这一样例中能够稳定完成候选范围控制、字段规划和报告生成，通常说明它已经具备较好的任务约束理解能力。

## 5.3 两组样例的联合观察重点

将上述两组样例放在一起观察时，可以重点比较以下几点：

- 系统是否能区分“有明确目标产品”和“有明确竞品约束”这两类任务；
- 在文档软件与会议软件两个场景下，动态字段是否会围绕实际任务重点发生变化；
- 对飞书和腾讯会议这两个关键对象，系统是否真正将其放在分析主轴上；
- 报告是否都能保持横向比较结构，而不是退化成产品资料汇总；
- 在后续追问或返工场景中，系统是否还能围绕原始任务约束继续工作。

如果这两组样例都能稳定跑通，基本可以说明系统已经能够覆盖当前项目里最核心的两类真实使用场景：

1. 围绕某个目标产品展开的竞品分析；
2. 带有用户指定竞品约束的行业竞品分析。

## 6. 示例输出应具备的关键特征

对于任一成功运行的样例，较理想的输出通常应体现以下特征：

1. 任务摘要清晰，能准确概括分析目标；
2. 候选竞品范围合理，且能区分直接竞品与替代竞品；
3. 分析字段既有通用核心维度，也能体现行业适配性；
4. 每个关键结论均能回到证据来源；
5. 报告结构清晰，可直接用于汇报或进一步修订；
6. 如存在质量问题，系统能够给出返工路径并改进结果；
7. 在报告之后，系统还可以继续生成问卷用于用户验证。

## 6.1 理想样例中应重点观察的内容

在实际查看样例时，建议特别关注以下几个方面：

- 系统是否只列出竞品名称，还是进一步区分了直接竞品与替代竞品；
- 动态 Schema 是否只是套用固定模板，还是结合行业特征补充了更有针对性的分析维度；
- 关键字段的分析是否建立在公开证据之上，而不是停留在抽象描述层面；
- QA 发现问题后，系统是否真正缩小到具体竞品、具体字段或具体章节继续返工；
- 报告是否支持继续追问和修订，而不是只能“看完即止”。

这些观察点的价值在于，它们能够帮助使用者区分“看起来很像完成了任务”和“实际上真正完成了任务”之间的差异。  
尤其是在多 Agent 系统中，表面上生成一份完整文本并不难，真正困难的是让每个阶段都留下可检查、可修正、可延展的痕迹。

## 7. 推荐查看方式

建议准备至少一组完整样例，并按以下顺序查看：

1. 输入任务描述
2. 自动识别行业与竞品范围
3. 动态生成分析 Schema
4. 多源采集与字段化证据沉淀
5. 字段级分析与报告生成
6. QA 检查与返工修正
7. 报告追问与内容修订
8. 问卷生成与问卷星导出

这种顺序能够更完整地体现系统区别于传统分析工具和一次性生成型 AI 工具的关键差异。

## 7.1 推荐同步查看的量化信息

建议在过程回放之外同步给出若干简单、直观的量化信息，例如：

- 本次任务覆盖的竞品数量；
- 规划的字段数量与最终完成分析的字段数量；
- 关键字段的平均证据条数；
- 返工前后的覆盖率变化；
- 最终 QA 问题数；
- 是否完成报告修订与问卷导出。

这些数字不需要很多，但能够帮助使用者更直观地理解系统改进了什么、完成到了什么程度。

## 8. 总结

本项目的评测应关注“任务是否真正完成、证据是否可追溯、结果是否可修正、结论是否还能继续延展”，而不是只看最终文本是否流畅。  
从整体价值看，多 Agent 协作、结构化证据沉淀和 QA 闭环修正机制，是本项目最核心的能力特点。
