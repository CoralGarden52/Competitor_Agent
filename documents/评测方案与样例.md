# 评测方案与样例

## 1. 文档概述

本项目的评测重点不只是“最终报告是否写得通顺”，而是整套竞品分析系统是否真正完成了任务。  
因此，评测时不仅要看最终文本，也要看它是否正确理解了任务、是否找到了合理的竞品、是否保留了证据来源、是否具备返工修正能力，以及结果是否还能继续被使用。

由于本项目属于多 Agent 协作系统，单纯比较一段最终文本很难完整反映它的能力，因此更适合采用“结构化指标 + 过程回放 + 人工质检”相结合的评估方式。

## 2. 评测目标

本项目评测主要围绕以下六类能力展开：

1. 能否正确理解用户任务并识别所属行业；
2. 能否自动发现合理的直接竞品与替代竞品；
3. 能否生成结构一致、可横向比较的分析 Schema；
4. 能否采集到来源明确、字段归属清晰的结构化证据；
5. 能否输出覆盖较完整、结论有依据的竞品分析报告；
6. 能否在 QA 发现问题后触发有效返工并改进结果。

## 3. 评测维度

## 3.1 任务理解与分析规划

这一维度重点评估系统是否能够把开放式任务转化为可执行分析计划。

**关注点**

- 行业识别是否合理
- 候选竞品是否覆盖主要直接竞品与替代竞品
- 动态 Schema 是否覆盖任务关心的核心问题
- 用户补充的竞品线索和关注点是否被吸收进规划结果

## 3.2 采集与证据质量

这一维度重点评估系统对公开信息的检索、筛选和组织能力。

**关注点**

- 证据来源是否多样
- 证据是否保留 URL 与来源信息
- 是否存在大量重复来源或明显无关内容
- 价格、功能、用户反馈等关键字段是否具备足够支撑
- 在证据不足时是否能触发更深入的补采和交叉核验

## 3.3 分析与报告质量

这一维度重点评估系统对结构化证据的归纳能力和最终报告质量。

**关注点**

- 字段覆盖是否完整
- 关键结论是否与证据一致
- 报告结构是否清晰稳定
- 是否能够形成横向比较，而不仅是逐家介绍
- 是否能够支持后续汇报、讨论和复用

## 3.4 质量闭环与修正能力

这一维度最能体现本项目与普通一次性生成型方案的差异。

**关注点**

- QA 是否能够识别证据不足、字段缺失和报告缺口
- 返工是否能定位到具体竞品、字段或章节
- 返工后覆盖率和结果质量是否提升
- 是否能够避免无意义的全链路反复重跑

## 3.5 结果延展能力

这一维度用于评估系统是否能够在报告之后继续发挥价值。

**关注点**

- 是否支持围绕报告继续追问和修订
- 是否支持将报告进一步转化为调研问卷
- 是否能形成“分析结论 -> 用户验证”的延伸链路

## 4. 评估方法

## 4.1 人工质检评估

由评审人员或使用者阅读系统输出，重点判断：

- 任务理解是否准确
- 竞品选择是否合理
- 证据是否可信
- 结论是否回应原始问题
- 报告是否可用于进一步汇报或决策参考

这种方式更适合判断系统在真实使用场景中的实际价值。

## 4.2 结构化指标评估

结合系统输出的结构化中间结果，可统计如下指标：

- 竞品覆盖率
- 字段覆盖率
- 单字段证据条数
- 来源站点多样性
- QA 打回次数
- 返工后覆盖率提升情况
- 报告追问后的补充有效性

这种方式更适合展示系统在“过程质量”上的工程化特征。

## 4.3 样例回放评估

对固定样例任务保存完整运行结果，并比较：

- 首轮结果
- QA 打回后的返工结果
- 最终报告
- 后续追问与问卷结果

这种方式很适合用于答辩演示，因为它可以同时展示系统的主链路和闭环修正能力。

## 5. 评测样例

## 5.1 样例一：通用 AI 智能体平台竞品分析

**输入示例**

```text
请分析通用 AI 智能体平台的主要竞品，重点关注产品能力、定价模式和用户反馈，并输出可用于汇报的结构化报告。
```

**预期表现**

- 系统能够识别行业为通用 AI Agent / 智能体平台
- 自动发现直接竞品与替代竞品
- 生成包含能力、价格、用户反馈、优势劣势等字段的动态 Schema
- 最终报告体现横向比较，而不是逐家孤立介绍

## 5.2 样例二：教育类 AI 产品竞品分析

**输入示例**

```text
调研 AI 教育助手相关产品，关注课程辅导能力、面向人群、收费方式和家长用户评价。
```

**预期表现**

- 系统能够识别教育行业场景
- 在通用字段外增加教育行业相关分析维度
- 在用户反馈部分兼顾官网信息与公开社区评价

## 5.3 样例三：SaaS 工具定价与功能对比

**输入示例**

```text
分析国内知识库/协作类 SaaS 工具的竞品情况，重点比较功能层级、付费模式和企业场景适配度。
```

**预期表现**

- 系统能够更细致地处理定价相关字段
- 结果中体现不同套餐、目标客群和企业适配能力差异
- 报告生成后仍支持继续追问和修订

## 6. 示例输出应具备的关键特征

对于任一成功运行的样例，较理想的输出通常应体现以下特征：

1. 任务摘要清晰，能准确概括分析目标；
2. 候选竞品范围合理，且能区分直接竞品与替代竞品；
3. 分析字段既有通用核心维度，也能体现行业适配性；
4. 每个关键结论均能回到证据来源；
5. 报告结构清晰，可直接用于汇报或进一步修订；
6. 如存在质量问题，系统能够给出返工路径并改进结果；
7. 在报告之后，系统还可以继续生成问卷用于用户验证。

## 7. 推荐展示方式

若用于比赛评审或课程答辩，建议准备至少一组完整样例，并按以下顺序展示：

1. 输入任务描述
2. 自动识别行业与竞品范围
3. 动态生成分析 Schema
4. 多源采集与字段化证据沉淀
5. 字段级分析与报告生成
6. QA 检查与返工修正
7. 报告追问与内容修订
8. 问卷生成与问卷星导出

这种展示顺序能够完整体现系统区别于传统分析工具和一次性生成型 AI 工具的关键差异。

## 8. 总结

本项目的评测应关注“任务是否真正完成、证据是否可追溯、结果是否可修正、结论是否还能继续延展”，而不是只看最终文本是否流畅。  
从整体价值看，多 Agent 协作、结构化证据沉淀和 QA 闭环修正能力，才是本项目最值得展示的核心特点。
