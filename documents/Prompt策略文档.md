# Prompt 策略文档

## 1. 文档概述

本项目并不是依靠“一条超长 Prompt 直接生成整份报告”来完成竞品分析。  
真正起作用的是一套分阶段、多 Agent 协作的机制：系统先理解任务，再采集信息、组织证据、形成字段级分析、完成质量审查，最后生成正式报告。

在这套机制里，Prompt 的作用不只是“让模型写点东西”，而是承担了任务分工、阶段约束、结构化输出和质量控制等更重要的职责。

本文件从系统设计角度说明本项目的 Prompt 策略，重点解释：

1. 为什么本项目采用分阶段 Prompt 设计；
2. 不同 Agent 的 Prompt 分别承担什么职责；
3. Prompt 如何与结构化输出、工具调用和 QA 机制协同工作；
4. Prompt 设计如何支撑系统的稳定性、可追溯性和可扩展性。

## 2. 总体策略

## 2.1 分阶段 Prompt，而非单轮大生成

竞品分析本身是一个长链路任务，至少涉及任务理解、竞品识别、信息采集、字段分析、质量检查、报告生成和后续修订等环节。  
如果使用单轮大 Prompt 直接要求模型完成全部任务，往往会出现上下文过长、信息焦点不稳定、结果结构不一致和证据无法追溯等问题。

因此，本项目将 Prompt 设计嵌入到分阶段工作流中，让每一类 Prompt 只服务于一种明确任务。这样做的好处是每一步都更聚焦，也更容易检查哪里做得好、哪里需要返工。

## 2.2 结构化输出优先

系统追求的不是“尽量长的自然语言输出”，而是后续阶段能够稳定读取和继续使用的结构化结果。  
因此，Prompt 会和 `JSON Schema`、结构化 handoff、字段级结果对象等机制配合使用，使不同 Agent 之间传递的信息清晰、稳定、可验证。

## 2.3 工具增强优先于凭记忆作答

在涉及竞品事实、价格、功能、用户反馈和公开评价时，系统并不依赖模型内部记忆直接作答，而是通过 Prompt 明确引导模型调用搜索、抓取和抽取工具，先获取证据，再形成分析。

## 2.4 Prompt 同时服务于生成与质控

本项目中，Prompt 不只用于规划、采集、分析和写作，也用于自评估、QA 检查和 deep dive 补采。  
换句话说，Prompt 不只是负责“生成内容”，也参与“判断质量”和“推动修正”。

## 2.5 Prompt 在本项目中的真实作用边界

在很多 AI 应用中，Prompt 更像是一段“让模型说话”的文字说明；但在本项目中，Prompt 的角色更接近一种任务协议。  
它首先要告诉模型“你当前所处的阶段是什么、你的职责边界是什么、你的输入是什么、输出必须长成什么样”，其次还要约束模型“哪些事情可以做、哪些事情不能做、什么时候应该调用工具、什么时候必须回到证据本身”。  
因此，本项目里的 Prompt 不是孤立存在的写作技巧，而是和状态、工具、Schema、QA 机制共同组成执行系统的一部分。

如果把系统比作一条完整的竞品研究流水线，那么 Prompt 更像是每一道工序的操作说明书。  
它并不直接替代搜索、抽取、状态管理和前端展示这些工程能力，但决定了模型在每个阶段会如何理解任务、如何组织信息、以及如何把结果传递给下游阶段。

## 2.6 为什么本项目不能只靠一个总 Prompt

从表面看，竞品分析似乎可以被描述成一句话，例如“请帮我分析某个行业的竞品，并输出一份报告”。  
但一旦进入真实使用场景，这个任务实际上同时包含了多个完全不同的问题：

- 需要先判断用户到底在研究哪个行业、哪个产品方向；
- 需要自动发现直接竞品、替代竞品和可能遗漏的参照对象；
- 需要围绕功能、价格、用户反馈等不同字段分别采集证据；
- 需要对证据进行归类、去噪、归纳和横向比较；
- 需要检查哪些地方证据不够、哪些地方结论不稳、哪些地方表达不清；
- 最后还要把这些中间产物写成适合阅读和汇报的报告。

这些问题的输入形式、输出形式、关注重点都不同。  
如果让一个总 Prompt 一次性处理所有问题，模型很容易在长上下文中混淆阶段目标，出现“前面在搜集事实，后面又开始自由发挥”“明明应该输出结构化字段，却直接写成长段散文”的情况。  
因此，本项目选择把 Prompt 拆开，让每个 Prompt 只负责一类问题，这样系统整体才更稳定。

## 3. Agent 级 Prompt 设计

## 3.1 任务规划 Agent

### 主要职责

- 理解用户任务
- 推断所属行业
- 自动发现直接竞品与替代竞品
- 形成动态分析 Schema 与后续执行计划

### Prompt 设计重点

- 不直接输出长篇竞品报告
- 优先输出行业判断、候选竞品、字段规划与执行建议
- 保证核心字段存在，同时允许根据行业特征补充动态维度
- 将开放式问题转化为可执行的结构化分析计划

### 设计价值

该类 Prompt 的作用是让系统从一开始就围绕“正确的竞品范围”和“正确的分析维度”展开，避免后续采集和分析偏题。

### 在本项目中的实际意义

规划阶段是整条链路里最容易被低估、但其实最关键的一步。  
如果这个阶段没有正确理解任务，后面所有阶段都会在错误方向上继续努力。  
例如，如果系统没有识别出用户真正关心的是“企业级能力”而不是“面向个人用户的使用体验”，那么后续采集和分析就会偏离重点；如果系统没有识别出某个关键替代竞品，最终报告就会显得视角过窄。

因此，这一类 Prompt 的目标不是立刻产出最终结论，而是先把问题空间框定好。  
它负责完成三个动作：第一，把开放式需求压缩成可理解的研究目标；第二，把竞品范围从模糊候选收敛为可分析对象；第三，把后续需要关注的字段提前规划出来。  
只有这样，后面采集、分析和写作阶段才有稳定的起点。

## 3.2 信息采集 Agent

### 主要职责

- 按“竞品 + 字段”组织采集
- 调用搜索、抓取和内容抽取能力
- 形成字段化证据集合

### Prompt 设计重点

- 明确当前采集目标对应哪个竞品、哪个分析字段
- 约束检索目标，减少无关网页和噪声内容
- 强调来源记录、字段归属和证据可追溯性

### 设计价值

该类 Prompt 使采集行为具有针对性，避免将大量无结构网页内容一次性送入模型，从而控制上下文长度并提升证据质量。

### 在本项目中的实际意义

采集阶段最难的问题不是“能不能搜到网页”，而是“搜到的信息能不能真正服务于后续分析”。  
如果采集 Prompt 不够明确，模型和工具往往会返回大量看似相关、实际上难以直接使用的内容，例如泛泛而谈的介绍页、重复转载的文章、与目标字段并不直接对应的营销文案。

本项目之所以强调“竞品 + 字段”的采集方式，就是因为不同字段关心的信息完全不同。  
价格字段更关注套餐、计费周期、版本差异；功能字段更关注官网产品描述和帮助文档；用户反馈字段则更需要社区讨论、评测文章或公开评价。  
因此，采集 Prompt 的职责不是简单“去搜一下”，而是让工具搜到尽可能接近目标字段的证据。

## 3.3 Deep Dive 补采 Agent

### 主要职责

- 在关键字段证据不足、来源过少或存在冲突时，进行补充检索与交叉核验

### Prompt 设计重点

- 聚焦单个竞品、单个字段或单个问题点
- 目标是补充证据和验证结论，而不是重新执行整轮大范围搜索
- 输出支持、冲突、缺口和验证状态

### 设计价值

该类 Prompt 使系统在关键字段上具备“二次求证”能力，是提高结论可信度的重要手段。

### 在本项目中的实际意义

Deep Dive 并不是“再多搜一轮”这么简单。  
它出现的前提是：系统已经完成了一轮基础采集，但仍然发现某些字段证据不足、来源过少，或者不同来源之间存在明显冲突。  
在这种情况下，如果还是按照常规全量搜索方式继续执行，往往会产生更多重复信息，而不一定真正补上关键缺口。

因此，Deep Dive Prompt 需要更聚焦。  
它要把问题缩小到一个明确单元，例如“某个竞品的定价字段仍然不完整”或“某个竞品的安全能力存在互相矛盾的说法”。  
只有在这种更细粒度的问题描述下，补采才有可能真正提高结果质量，而不是单纯扩大信息量。

## 3.4 字段分析 Agent

### 主要职责

- 对同一竞品、同一字段下的证据进行归纳分析
- 输出字段级摘要、归一化结果、证据缺口和置信度

### Prompt 设计重点

- 一次只分析一个字段
- 所有结论都必须建立在当前证据集合上
- 输出结构保持统一，便于横向对比和报告写作

### 设计价值

该类 Prompt 支撑系统采用“字段级分析 -> 报告级写作”的两段式生成架构，使中间结果能够复用并保持稳定。

### 在本项目中的实际意义

字段分析阶段承担的是“把证据真正转化为可比较结论”的工作。  
如果系统直接从原始网页内容生成整份长报告，模型很容易一边看事实、一边组织篇章、一边做价值判断，导致不同层次的信息混在一起，最终既难以追溯，也难以返工。  
而字段级 Prompt 的好处在于，它只要求模型解决一个更小、更明确的问题，例如“针对某个竞品总结定价模式”或“基于当前证据归纳用户反馈特征”。

这种设计让系统能够先积累稳定的中间分析产物，再在后续阶段把这些产物拼装成完整报告。  
从工程角度看，这比直接写长文更稳；从业务角度看，这也让报告更容易被修订，因为修改者不需要每次都回到原始网页重新梳理全部信息。

## 3.5 QA 审查 Agent

### 主要职责

- 检查覆盖度、证据充分性、分析一致性和报告完整性
- 判断问题应回到采集、分析还是写作阶段

### Prompt 设计重点

- 不仅判断是否通过
- 需要指出问题属于哪个阶段、哪个竞品、哪个字段或哪个章节
- 尽量形成可执行的返工方向，而非笼统重做

### 设计价值

该类 Prompt 让 QA 不再是简单的结果判定器，而是闭环修正机制中的关键反馈节点。

### 在本项目中的实际意义

QA 阶段如果只是回答“通过”或“不通过”，其实解决不了太多问题。  
真正有价值的 QA Prompt，必须能够进一步指出问题到底发生在哪里，以及下一步应该往哪个方向修。  
例如，某个字段结论不稳，可能是因为证据还不够；某一章节写得空泛，可能是因为分析阶段没有形成足够稳定的中间结果；某个竞品缺失，可能是规划阶段就漏掉了范围。

因此，本项目里的 QA Prompt 更像是一位研究复核助手，而不是简单的自动判定器。  
它既要检查结果，也要把问题转化成后续可执行的返工任务。  
这也是系统能够实现“发现问题 - 定位问题 - 修复问题”闭环的重要原因。

## 3.6 报告写作 Agent

### 主要职责

- 基于结构化分析结果生成正式 Markdown 报告
- 支持流式输出长报告

### Prompt 设计重点

- 以中间分析结果为输入，而不是直接重新阅读所有原始证据
- 强调章节稳定、逻辑清晰和可读性
- 保留分析依据和结构一致性

### 设计价值

该类 Prompt 使报告生成建立在结构化中间结果之上，显著降低长链路自由生成带来的漂移问题。

### 在本项目中的实际意义

报告写作阶段的 Prompt 不承担“重新研究一遍竞品”的职责。  
它更像是在已有研究结果之上完成表达组织、重点提炼和章节安排。  
这意味着写作 Prompt 面对的输入，并不是原始网页，而是已经经过筛选和归纳的字段级中间结果。

这样做有两个好处。  
第一，报告结构会更稳定，因为模型不需要一边理解原始证据，一边临时决定如何写作。  
第二，后续修订成本更低，因为报告生成与分析生成已经解耦，改写表达时不必重新触碰整套分析链路。

## 3.7 报告追问 Agent

### 主要职责

- 在报告生成后继续回答问题、补充说明和修订内容

### Prompt 设计重点

- 判断当前问题属于解释、补充、改写还是事实查询
- 必要时继续调用工具补充证据
- 结合报告分块和工作记忆控制上下文长度

### 设计价值

该类 Prompt 使系统具备持续协作能力，而不是把报告当作一次性交付物。

### 在本项目中的实际意义

真实业务中的报告几乎不会一稿定稿。  
用户往往会继续追问“这个判断依据是什么”“请把这一段改得更适合汇报”“再补一个竞品的说明”“把这部分写得更强调价格差异”。  
如果系统不能处理这些后续问题，那么它的价值就会停留在第一次生成上。

报告追问阶段的 Prompt 之所以重要，是因为它决定了系统能否在已有成果之上继续协作。  
这一阶段不是重新开启一轮全量分析，而是要基于既有报告、工作记忆和必要补检索来继续推进。  
它服务的是“持续完善结果”这一需求，而不是“重新生成一份新文本”。

## 3.8 问卷生成 Agent

### 主要职责

- 从竞品分析报告中提取适合调研的问题信号
- 生成可直接用于用户调研的问卷内容

### Prompt 设计重点

- 关注可转化为用户问题的分析结论
- 避免暴露内部分析术语和系统中间结构
- 通过 review 机制检查问卷自然度、清晰度和可回答性

### 设计价值

该类 Prompt 让系统从“形成分析结论”延伸到“验证分析结论”，增强产品化价值。

### 在本项目中的实际意义

从产品角度看，问卷生成阶段是系统区别于普通竞品报告工具的重要延展。  
很多分析工作在形成报告后就结束了，但真实业务里，团队往往还需要进一步验证这些判断是否符合真实用户体验。  
例如，报告里认为某一竞品的价格门槛高、某一功能更受欢迎、某一使用场景更具优势，那么这些结论都可以继续转化为调研问题。

因此，问卷生成 Prompt 的职责不是简单改写报告，而是挑选其中最适合进入用户验证环节的结论，并把它们转换成自然、可回答、不会暴露内部分析术语的问题。  
只有这样，问卷链路才真正发挥出“从公开研究走向用户验证”的作用。

## 4. 关键设计原则

## 4.1 明确职责边界

每一类 Prompt 都只服务于一个明确阶段，避免角色越权和任务混乱。这种设计对于多 Agent 系统尤为重要。

## 4.2 以证据为中心

系统中的 Prompt 普遍强调“证据先行，结论后置”，确保模型输出建立在公开可核验信息之上。

## 4.3 以结构为中心

Prompt 的输出与结构化字段紧密耦合，使后续阶段可以稳定读取，而不是依赖脆弱的自然语言解析。

## 4.4 支持返工与迭代

Prompt 并不只面向“首次成功生成”，也面向“被 QA 打回后的局部修复”。  
在返工场景下，Prompt 会收敛到更细粒度的问题单元，例如特定竞品、特定字段或特定章节。

## 4.5 兼顾通用性与行业适配

Prompt 一方面保留核心竞品分析框架，另一方面允许结合行业特征和用户关注点生成动态字段，提升跨行业任务适应性。

## 4.6 与字段化证据机制协同

本项目的 Prompt 设计并不是孤立存在的，而是与“竞品 + 字段”的证据组织方式紧密结合。  
在采集阶段，Prompt 会明确当前要解决的是哪个竞品、哪个字段的问题；在分析阶段，Prompt 则只围绕该字段对应的证据集合进行归纳，而不会重新阅读整份原始网页集合。

这种设计有两个直接价值：

- 控制上下文长度，避免因网页内容过多而造成推理分散；
- 让每个阶段的输出都更容易回到具体证据和具体字段，增强可追溯性。

## 4.7 与 QA 返工机制协同

Prompt 的价值不仅体现在“第一次怎么生成”，也体现在“出现问题后如何修复”。  
当 QA 发现某些竞品、字段或章节存在问题时，后续 Prompt 不会再面向整条链路重新执行，而是收敛到更小粒度的返工目标，例如：

- 针对某个竞品继续补采关键证据；
- 针对某个字段重新组织分析；
- 针对某个章节重新改写表达结构。

这种局部返工式 Prompt 设计，使系统在出现问题时能够更有针对性地继续修正。

## 4.8 与自评估机制协同

系统在规划、采集、分析和草稿生成等关键阶段都引入了自评估信号。  
Prompt 在这里承担的作用，不只是要求模型给出内容，还要求模型对当前结果的覆盖度、一致性、证据质量和不确定性进行内部判断。

这种做法的意义在于，系统能够在最终 QA 之前就提前识别高风险中间结果，减少低质量内容继续传递到下游阶段的概率。

## 5. Prompt 与系统其他机制的协同关系

### 5.1 与工作流编排协同

Prompt 的作用始终嵌入在状态驱动工作流中，随着阶段变化而变化，并不独立运行。

这意味着 Prompt 不会脱离上下文单独发挥作用。  
同一个模型，在规划阶段面对的是任务描述和候选线索，在分析阶段面对的是字段化证据，在写作阶段面对的是结构化中间结果，在 QA 阶段面对的是待检查的分析与报告内容。  
Prompt 只有放回工作流里，才能真正理解它为什么被这样设计。

### 5.2 与工具调用协同

Prompt 用于明确何时检索、何时抽取、何时依赖证据归纳，帮助模型与工具形成分工。

如果没有这一层分工，模型就很容易陷入“明明应该先找证据，却直接凭经验给答案”的状态。  
本项目的 Prompt 通过明确告诉模型当前阶段应该做什么、何时需要借助工具、输出必须依据哪些信息，使工具和模型之间形成更清晰的合作关系。

### 5.3 与 QA 闭环协同

Prompt 不仅决定如何生成，也决定如何检查和修复，使系统具备可持续优化能力。

对于本项目而言，Prompt 最有价值的地方之一，就是它不是只服务于“成功生成第一次结果”，而是同时服务于“结果不够好时怎么继续修”。  
这也是为什么 Prompt 文档必须和 QA、返工、补采放在一起说明，而不能把它单独理解成写作模板。

### 5.4 与动态 Schema 协同

Prompt 在规划阶段参与生成分析字段，在采集和分析阶段又围绕这些字段继续组织信息，使系统在结构上保持一致。

动态 Schema 机制让系统具备跨行业适配能力，而 Prompt 则负责把这种结构真正落实到每一个阶段。  
没有 Prompt 的约束，动态字段很容易只停留在“规划出来了”，却没有真正进入后续采集、分析和写作过程。  
正是因为 Prompt 持续围绕这些字段组织输入和输出，动态 Schema 才成为一条真正有效的执行线索。

### 5.5 与多轮报告修订协同

报告生成之后，Prompt 仍然继续发挥作用。  
在报告追问与修订阶段，系统需要判断用户是在要求：

- 解释已有结论；
- 补充缺失信息；
- 改写表达方式；
- 进一步搜索某个事实点。

因此，这一阶段的 Prompt 更强调“基于既有成果继续协作”，而不是重新生成整份报告。这也是系统能够支持持续修订的重要前提。

### 5.6 与问卷生成协同

在问卷生成阶段，Prompt 的目标已经从“分析竞品”转变为“提炼适合用户调研的问题”。  
这一阶段会重点关注：

- 哪些结论适合进一步验证；
- 哪些比较项可以转化为用户可理解、可回答的问题；
- 如何避免暴露系统内部分析术语与中间结构。

这使系统能够自然地从研究结论过渡到用户验证环节，而不是简单复制报告内容。

## 6. 总结

对于本项目而言，Prompt 的价值不在于“把一句话写得更漂亮”，而在于把复杂的竞品分析任务拆成多个可执行、可检查、可返工的阶段。  
这种设计让系统既能保持生成能力，又能兼顾稳定性、可追溯性和持续修正能力，更适合真实的竞品研究场景。

从整体设计看，Prompt 在本项目中更像一种“任务组织与质量约束机制”，而不仅仅是文本生成指令。  
也正因为如此，Prompt 才能与多 Agent 协作、结构化证据、QA 返工、自评估和问卷延展共同组成一套完整的竞品分析方案。

本项目的 Prompt 策略并不是通过“一个很强的总 Prompt”解决所有问题，而是通过分阶段、结构化、证据优先和可返工的 Prompt 设计，把模型能力真正嵌入到工程系统中。  
这种设计方式更适合竞品分析这类需要多步推理、事实支持、结构统一和持续修订的复杂任务。

如果只把 Prompt 理解成“模型前面那段文字提示”，就很难理解本项目的设计重点。  
更准确的说法是：Prompt 在这里承担的是一套研究流程的行为约束和阶段协议。  
它帮助系统在不同阶段始终围绕正确的问题工作，帮助中间结果保持结构稳定，也帮助质量问题在出现后能够被明确识别并继续修正。

因此，这份 Prompt 策略文档真正想说明的是：本项目的智能能力不是建立在某个单点技巧上，而是建立在一套分工清晰、证据优先、支持返工、能够延展到后续验证的整体设计之上。  
这也是本项目能够从“会生成内容”走向“能完成复杂竞品研究任务”的重要原因。
