# Frontend 项目实施方案

## 1. 项目目标

前端目标是把竞品分析智能体系统做成一个可交互、可解释、可演示的工作台。用户不只是看到一份最终报告，还能看到：

1. 任务如何发起。
2. 多个智能体如何分阶段协作。
3. 证据、分析、报告、质检结果如何逐步形成。
4. 报告生成后如何继续追问、修订和回看。
5. 系统异常、重试、返工和完成状态如何对用户呈现。

## 2. 总体建设思路

前端建设不采用“静态报告页 + 一个聊天框”的简单形态，而采用“工作区 + 实时状态流 + 报告追问 + 证据回放”的方式分阶段推进。

整个实施分为五个阶段：

1. 先搭建页面框架与核心状态模型。
2. 再打通任务创建、运行状态、工作区快照展示。
3. 再做多 Agent 可视化、事件流和报告查看。
4. 再做报告追问、多轮交互与流式输出。
5. 最后补异常处理、断流恢复、交互细节和演示打磨。

在实施过程中，编码助手主要负责状态流梳理、组件重构建议、接口接入草拟、类型补齐和联调问题定位；开发者负责交互取舍、页面层级、信息密度和最终体验判断。

## 3. 分阶段实施计划

### 阶段一：页面骨架与状态模型

本阶段的重点是把工作台骨架搭好，让后续能力有稳定落点。

建设内容：

- 设计首页、历史记录区、工作区区块和报告区块的页面结构。
- 定义前端核心状态：
  - 当前任务输入
  - 当前 run/workspace 数据
  - 事件流数据
  - 聊天消息
  - 报告编辑状态
  - 问卷与导出状态
- 明确“单次任务执行”和“报告追问对话”是两条不同的交互链路。

实施方式：

- 开发者先画出工作区的信息层级和用户主路径。
- 编码助手辅助把状态划分到合适的 hooks、refs 和工具函数中。
- 第一轮先保证信息结构清楚，不急于做复杂动效。

### 阶段二：基础接口接入

本阶段先把前后端最小闭环跑通。

建设内容：

- 接入创建任务接口。
- 接入 run 状态接口与 workspace 快照接口。
- 在页面上展示阶段卡片、摘要信息和初始报告。
- 保证用户可以切换历史会话并恢复上次工作区。

实施方式：

- 开发者负责确认哪些数据必须首屏展示。
- 编码助手协助生成类型、请求封装和状态同步逻辑。
- 每接一个接口，都先做最小页面验证，避免一口气堆太多状态导致难排错。

### 阶段三：多智能体过程可视化

前端的价值不只是“展示结论”，还要把中间过程做得足够清楚。

建设内容：

- 展示各阶段状态、执行顺序、阶段输入输出摘要。
- 展示事件流列表和阶段内关键事件。
- 展示证据、schema、handoff、trace 等工作区细节。
- 让用户能看见系统不是单模型一次性吐答案，而是多步骤协作完成。

实施方式：

- 先按阶段组织信息，再逐步补细节面板。
- 编码助手辅助从接口结构中提炼 UI 所需映射关系。
- 开发者重点判断信息是否过多、哪些字段适合默认收起、哪些值得高亮显示。

### 阶段四：报告追问与流式交互

本阶段是产品体验上最容易形成亮点的部分。

建设内容：

- 在已有报告基础上支持继续提问。
- 区分“任务主消息流”和“报告追问消息流”。
- 让报告追问支持实时进度更新和正文逐步输出。
- 在必要时支持报告更新后同步刷新工作区。

实施方式：

- 第一阶段先实现异步 turn 模式，用户提交后可看到处理中状态。
- 第二阶段升级为 SSE 流式模式：
  - 先接入阶段事件
  - 再接入正文增量
  - 最后处理最终结果与本地 pending message 的对齐
- 编码助手在这一阶段主要负责：
  - 分析消息状态流
  - 生成 `EventSource` 接入代码
  - 处理 snapshot、delta、done 等事件
  - 协助联调后修正边界条件

### 阶段五：稳定性与演示细节

最后一阶段重点是提升使用稳定性和展示完成度。

建设内容：

- SSE 中断、重连、回退策略。
- 历史消息和当前工作区的一致性维护。
- 错误提示、加载态、禁用态等边界体验。
- 文案收敛、布局细节、视图切换顺滑度。

实施方式：

- 开发者根据真实演示场景确定哪些异常必须兜住。
- 编码助手帮助扫描潜在竞态、补清理逻辑和重连逻辑。
- 每个主要交互都配一条“从开始到完成”的手测脚本。

## 4. 协作方式

前端开发采用“先拆问题，再增量落地”的协作方式，而不是一句 prompt 生成整页。

协作节奏如下：

1. 开发者先明确当前要解决的是哪条用户路径。
2. 编码助手先分析现有状态流、受影响组件和最小改造范围。
3. 双方确认后再进行局部改动。
4. 每次改动都做一次快速联调或构建验证。
5. 体验不理想时继续迭代，而不是为了保持生成结果而勉强接受。

这种方式尤其适合本项目，因为前端同时面对：

- 运行中的主工作流
- 报告追问多轮消息
- 工作区快照刷新
- 实时事件流
- 历史会话恢复

这些状态如果一次性混在一起生成，很容易出现隐性 bug。

## 5. 关键里程碑

### M1：工作台骨架完成

验收标准：

- 能发起任务
- 能进入工作区
- 能看到历史会话

### M2：多 Agent 工作区可视化完成

验收标准：

- 能看见阶段卡片
- 能看见事件流
- 能查看报告和证据

### M3：报告追问基础能力完成

验收标准：

- 能对已有报告继续提问
- 能看到回答结果
- 能回查历史追问记录

### M4：报告追问流式输出完成

验收标准：

- 提问后能看到处理中状态
- 回答正文能逐步输出
- 完成后最终消息能与持久化结果对齐

### M5：异常恢复与演示细节完成

验收标准：

- SSE 中断后界面不失控
- 切换历史会话状态稳定
- 错误提示清楚
- 页面可用于现场演示

## 6. 风险与应对

### 风险一：状态源太多，消息和工作区容易不同步

应对方式：

- 明确主数据源与派生数据源
- 把 run 级状态流和 report chat 状态流区分开
- 最终结果统一回落到持久化接口数据

### 风险二：流式交互体验不稳定

应对方式：

- 保留最终结果查询接口作为兜底
- 对 snapshot、delta、done 分别定义职责
- 在前端维护清晰的 pending message 生命周期

### 风险三：信息过多导致工作区难用

应对方式：

- 先保证“主路径信息”突出
- 把 trace、events、schema、依据等详细信息分层收起
- 通过默认选中阶段和重点区域高亮控制信息密度

## 7. 最终交付

前端最终交付内容包括：

- 工作台页面与历史记录页
- 多 Agent 协作过程可视化
- 工作区、报告、证据和事件流展示
- 报告追问与流式交互
- 稳定的错误处理与状态恢复
- 可支撑演示的交互说明与实施文档

## 8. 结语

前端建设的关键不是把数据“摆上去”，而是把智能体系统的运行过程、证据链路、返工闭环和追问体验组织成一个用户能看懂、能操作、能信任的工作台。  
整个实施过程适合采用“开发者把控交互目标，编码助手参与状态设计、局部重构与联调修正”的方式推进，这样更容易在复杂交互场景下保持稳定性和一致性。
