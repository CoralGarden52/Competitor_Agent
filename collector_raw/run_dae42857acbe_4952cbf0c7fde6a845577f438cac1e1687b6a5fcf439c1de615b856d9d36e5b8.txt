 OpenClaw作为一个 local-first 的 AI agent 平台,最近越来越受关注。我想把它和看板工具结合起来,构建一种“卡片驱动”的日常工作流:每一张卡片代表一个工作项(GitHub Issue 或 PR),而看板的每一列则充当一个状态机,驱动 AI agent 自动完成分析、规划和执行,人类只在关键审批点介入。 本文总结的是一套已经验证可行的集成方式:用Trello作为看板,用 OpenClaw 作为编排运行时,用GitHub Copilot CLI作为真正负责自主执行的 agent。 之所以这样分层,还有一个很现实的原因:OpenClaw 当前自带的 Agent 在处理长上下文自主任务时仍然比较吃力,因此这里选择使用 GitHub Copilot CLI 作为全自主执行层。 这个系统的核心价值主张很简单:Agent 完成 95% 的工作,人类做出 100% 的决策。Agent 负责所有高劳动密度的事情——克隆仓库、阅读代码、复现问题、搜索文档、编写修复、运行 CI,以及在 CI 失败时耐心重试(单次失败最多进行 9 次结构化修复尝试)。而人类只在高杠杆决策点介入:阅读简报、批准方案、合并 PR,而且这些操作都只需要在看板上拖动卡片。看板的列编码了一套严格的“Agent 提方案,人类做裁决”的协作契约;与此同时,人类始终保留一个即时止损开关——只要把卡片从In action列拖走,Agent 就必须停止,并回滚代码状态。最终效果是:一个维护者可以同时监督几十个 Issue 和 PR,把原本需要几个小时亲手处理的工作压缩成几分钟的决策时间。 Phase 0:Webhook 如何安全到达本地 Agent 从高层看,交付链路是这样的: Trello →Cloudflare Tunnel→ 验签网关 → 本地 Agent 入口 在任何 Agent 逻辑开始之前,网关会先做两件保护性的事情: 校验Trello 的X-Trello-Webhook签名,因此即使 Tunnel URL 泄露了,外部也不能轻易伪造请求。 把payload 裁剪到只剩事件类型、卡片 ID、列表迁移等路由字段,避免任意卡片文本直接进入 Agent prompt。 因此,OpenClaw 接收到的不是原始用户内容,而是一个经过认证、且已裁剪的路由事件。 Phase 1:Router 什么时候唤醒 Copilot 每个webhook 都会先被标准化成一个简短的事件字符串,例如: event=updateCard card_id=<card_id> list_before=Need Attention list_after=Analyze 随后Router Agent 会按顺序应用一个简单的状态机: 跳过等待人工的状态:如果卡片被移动到Need Attention、Pending PR、Ready for plan review或Ready for review,则直接停止。 跳过Agent 自己发的评论:如果最近一次commentCard评论以[agent]:开头,则直接停止。
