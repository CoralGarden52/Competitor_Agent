# 竞品分析报告生成技能

## 概述

本技能用于生成跨行业、跨产品的标准化竞品分析报告。基于公开互联网信息采集和多源证据交叉验证，输出结构化的 SWOT 分析报告与竞品对比矩阵，适用于产品经理、商业分析、战略决策场景。

## 核心能力

1. **跨行业适配**：支持 BI 工具、SaaS、AI 平台、云计算、CRM、金融科技、电商、制造业、医疗等任意行业的竞品分析。
2. **动态维度扩展**：在固定核心字段之外，根据行业特征自动生成动态分析维度（schema 字段），确保分析深度和行业针对性。
3. **标准化输出**：统一的 SWOT 分析框架 + 竞品对比矩阵 + 分章节结构化报告。
4. **证据可追溯**：每个结论必须关联到具体来源 URL 和证据 ID，确保报告可信度和可审计性。

## 输入规范

### 必需输入

| 参数 | 类型 | 说明 |
|------|------|------|
| `user_prompt` | string | 用户的原始分析请求，包含分析目标、关注点 |
| `competitors` | list[string] | 竞品名称列表（至少 2 个） |
| `industry` | string | 行业标签，如 "bi_tools"、"crm"、"cloud_computing" |

### 可选输入

| 参数 | 类型 | 说明 |
|------|------|------|
| `competitor_hints` | list[string] | 补充竞品线索 |
| `aspect_hints` | list[string] | 用户特别关注的维度提示词（会自动生成扩展 schema 字段） |
| `language` | string | 报告语言，默认 "zh" |
| `timeframe` | string | 分析时间范围 |

## 固定核心字段（Core Schema）

每个竞品分析**必须**覆盖以下 5 个固定维度 + 产品标识字段。这些字段构成竞品对比矩阵的基准列：

### 1. `product` — 产品名称

- **中文标签**：产品
- **类型**：string（对比矩阵的行键）
- **说明**：竞品产品/服务的正式名称，作为对比矩阵的行标识。可附加竞争关系标注（直接竞品/间接竞品）。
- **采集查询模板**：`{product}` 官网
- **推荐来源**：官网首页、产品介绍页

### 2. `feature_tree` — 功能体系

- **中文标签**：功能体系
- **类型**：tree / dict
- **normalized_value 结构**：
  ```json
  {
    "nodes": [
      {"name": "核心能力模块名称", "capability": "具体能力描述"}
    ]
  }
  ```
- **说明**：产品的核心功能树/能力矩阵，覆盖主要功能模块、技术架构、集成能力、使用场景。应以层级化方式呈现。
- **采集查询模板**：
  - `{product} 核心功能`
  - `{product} 官方文档 功能`
  - `{product} 使用场景`
- **推荐来源**：官网、产品文档、功能页、帮助中心
- **分析聚焦**：提取核心能力、功能结构、主要使用场景。

### 3. `strengths` — 优势

- **中文标签**：优势
- **类型**：list[string]
- **normalized_value 结构**：
  ```json
  {
    "items": ["优势点1", "优势点2", "优势点3"]
  }
  ```
- **说明**：产品的差异化优势、核心竞争力、相比竞品的突出表现。应基于评测和社区反馈提炼，避免仅复述官方宣传。
- **采集查询模板**：
  - `{product} 优势 评测`
  - `{product} 对比 缺点`
  - `{product} 为什么选择`
- **推荐来源**：第三方评测、技术分析、社区讨论
- **分析聚焦**：总结差异化优势与核心竞争力。

### 4. `weaknesses` — 劣势

- **中文标签**：劣势
- **类型**：list[string]
- **normalized_value 结构**：
  ```json
  {
    "items": ["劣势点1", "劣势点2", "劣势点3"]
  }
  ```
- **说明**：产品的局限性、短板、已知问题、用户抱怨。应基于真实用户反馈和独立评测，避免主观臆断。
- **采集查询模板**：
  - `{product} 劣势 局限`
  - `{product} 对比 缺点`
  - `{product} 缺点 评测`
- **推荐来源**：评测、社区反馈、问题追踪
- **分析聚焦**：总结局限、风险与常见担忧。

### 5. `pricing_model` — 定价模式

- **中文标签**：定价模式
- **类型**：object
- **normalized_value 结构**：
  ```json
  {
    "model_type": "subscription|usage_based|enterprise_quote|freemium|one_time|unknown",
    "free_tier": true|false,
    "billing_dimensions": ["seat", "usage", "storage", "api_calls"],
    "tiers": [
      {
        "name": "套餐名称",
        "price_range": "价格范围（含货币单位）",
        "billing_cycle": "monthly|annual|one_time",
        "limits": ["限制条件"]
      }
    ]
  }
  ```
- **说明**：产品的完整定价信息，包括定价模式类型、是否有免费层、计费维度、各套餐/版本的价格和限制。商业决策的关键维度，应尽可能精确。
- **采集查询模板**：
  - `{product} {year} 价格 套餐 元/月`
  - `{product} {year} 企业版 价格 元/年`
  - `{product} {year} 收费 版本 对比 元/人/月`
- **推荐来源**：官网定价页、产品文档、第三方评测
- **分析聚焦**：分析定价结构、套餐层级、计费维度和免费策略。

### 6. `user_feedback` — 用户反馈

- **中文标签**：用户反馈
- **类型**：object
- **normalized_value 结构**：
  ```json
  {
    "positive_themes": ["正面反馈主题1", "正面反馈主题2"],
    "negative_themes": ["负面反馈主题1", "负面反馈主题2"],
    "representative_quotes": ["代表性用户引言"],
    "sentiment_distribution": {
      "positive": 0.6,
      "neutral": 0.25,
      "negative": 0.15
    }
  }
  ```
- **说明**：基于公开渠道（社区、评测、社交媒体）的用户真实反馈汇总。需区分正面和负面主题，提供代表性引言，给出情感分布估算。
- **采集查询模板**：
  - `{product} 知乎 评价`
  - `{product} 点评`
  - `{product} 体验`
  - `{product} 反馈`
- **推荐来源**：知乎、专业社区、产品评测、应用商店评论
- **分析聚焦**：提取正负面主题和代表性用户反馈。

## 动态 Schema 字段（Dynamic Fields）

除固定核心字段外，系统会根据**行业特征**和**用户 aspect_hints** 自动生成扩展分析维度。

### 动态字段生成规则

1. **数量限制**：动态字段总数控制在 5-7 个（含核心字段），即动态扩展字段为 0-2 个。
2. **行业驱动**：根据 `industry` 参数推断行业特有维度。例如：
   - **BI 工具行业**：`data_security`（数据安全）、`mobile_support`（移动端支持）、`ai_capability`（AI 分析能力）
   - **SaaS/CRM 行业**：`integration_ecosystem`（集成生态）、`customization`（定制化能力）、`sla_guarantee`（SLA 保障）
   - **云计算行业**：`global_regions`（全球区域覆盖）、`compliance_certs`（合规认证）、`performance_benchmark`（性能基准）
   - **AI 平台行业**：`model_variety`（模型丰富度）、`fine_tuning_support`（微调支持）、`inference_latency`（推理延迟）
   - **金融科技行业**：`regulatory_compliance`（监管合规）、`risk_control`（风控能力）、`transaction_speed`（交易速度）
3. **用户驱动**：当用户提供 `aspect_hints` 时，直接将提示词转换为 schema 字段并合并到分析计划中。
4. **优先级**：动态字段的 priority 设置为 100+，排在核心字段之后。

### 动态字段的 normalized_value 通用结构

```json
{
  "key_observations": ["关键观察点1", "关键观察点2"],
  "value": "该维度的核心结论"
}
```

### 动态字段的采集模板

- `{product} {field_name}`
- `{product} {field_name} 对比`

## 竞品分类体系

每个竞品在分析中应标注其竞争关系类型：

| 类型 | 标识 | 说明 | 对比矩阵显示 |
|------|------|------|--------------|
| **直接竞品** | `direct` | 同一赛道、相似目标用户、可替代关系 | 产品名（直接竞品） |
| **间接竞品** | `substitute` | 不同赛道但有功能重叠或场景替代性 | 产品名（间接竞品） |

竞品分类决策应基于：
1. 目标市场和用户群重叠度
2. 核心功能相似度
3. 定价和商业模式可比性
4. 用户认知中的替代关系

### 竞品对比矩阵（Comparison Matrix）

对比矩阵是报告的核心输出之一，格式要求：

```markdown
| 产品 | 功能体系 | 优势 | 劣势 | 定价模式 | 用户反馈 | [动态字段1] | [动态字段2] |
|------|---------|------|------|---------|---------|-------------|-------------|
| 产品A（直接竞品） | ... | ... | ... | ... | ... | ... | ... |
| 产品B（直接竞品） | ... | ... | ... | ... | ... | ... | ... |
| 产品C（间接竞品） | ... | ... | ... | ... | ... | ... | ... |
```

**矩阵规则**：
1. 第一列固定为 `product`（产品名称），后续列为所有 schema 字段。
2. 核心字段在前，动态字段在后，按 priority 排序。
3. 每个单元格应为该字段的 summary 精简版（≤160 字符）。
4. 当某字段证据不足时填入 `"需进一步确认"`，而非留空。
5. 矩阵中的每个结论都应可追溯到 evidence_refs。

### SWOT 分析输出

```json
{
  "swot": {
    "strengths": [
      "跨竞品共同优势或我方产品相对优势",
      "具体应引用 findings 中 category=feature 的高置信度结论"
    ],
    "weaknesses": [
      "跨竞品共同短板或我方产品相对劣势",
      "具体应引用 findings 中 category=risk 和字段 weaknesses 的结论"
    ],
    "opportunities": [
      "基于竞品劣势识别的市场机会",
      "基于用户反馈中未满足需求的机会点",
      "基于动态维度差异化的机会"
    ],
    "threats": [
      "竞品优势带来的竞争威胁",
      "行业趋势变化带来的风险",
      "证据覆盖不足带来的决策风险"
    ]
  }
}
```

**SWOT 编写规则**：
1. SWOT 的每一项必须是**可执行的判断**，不是证据的简单罗列。
2. 每个条目应能追溯到 1 个以上的 evidence_refs 或 findings。
3. Opportunities 和 Threats 应结合跨行业的宏观视角。
4. Strengths 和 Weaknesses 应同时覆盖核心字段和动态字段。

## 动态字段在对比矩阵中的处理

当分析计划中包含动态 schema 字段时：

1. **矩阵列扩展**：每个动态字段作为对比矩阵的额外列，排在核心字段之后。
2. **报告章节扩展**：每个动态字段生成独立的报告章节，标题格式为 `"动态维度：{字段中文标签}"`，插入在"七、核心优劣势与风险"中。
3. **normalized_value 展示**：动态字段的 `key_observations` 以项目符号形式展示在对应章节中。
4. **SWOT 融合**：动态字段的分析结果应融合进 SWOT 的 Opportunities 和 Threats 中。
5. **字段标签映射**：动态字段名自动进行中文化，下划线转空格，若字段名包含中文则保留原样。

## 跨行业适配指南

### 行业分析差异化重点

| 行业 | 重点关注维度 | 推荐动态字段 | 定价分析侧重 |
|------|-------------|-------------|-------------|
| **BI/数据分析** | 可视化能力、数据源兼容、AI 辅助、中国式报表 | `data_security`, `mobile_support` | 按用户/按年、免费层策略 |
| **SaaS/CRM** | 集成生态、定制化、移动端、自动化工作流 | `integration_ecosystem`, `api_openness` | 按席位/月、企业版溢价 |
| **云计算/IaaS** | 全球节点、性能基准、SLA、合规认证 | `global_regions`, `compliance_certs` | 按需/预留实例、流量费 |
| **AI/ML 平台** | 模型丰富度、推理性能、微调能力、数据隐私 | `model_variety`, `inference_latency` | 按 token/调用次数 |
| **金融科技** | 监管合规、风控能力、交易速度、数据安全 | `regulatory_compliance`, `risk_control` | 按交易量/订阅 |
| **电商/SaaS** | 店铺管理、营销工具、物流集成、数据分析 | `marketing_tools`, `logistics_integration` | 按 GMV 抽成/月费 |
| **医疗健康** | HIPAA/GDPR 合规、互操作性、临床决策支持 | `regulatory_compliance`, `interoperability` | 按机构/年订阅 |
| **教育科技** | LMS 集成、内容创作、评估工具、多语言 | `lms_integration`, `assessment_tools` | 按学生/学期 |

### 跨行业通用原则

1. **定价模式**始终是决策关键维度，无论什么行业都应尽力采集精确信息。
2. **用户反馈**应从行业垂直社区获取（如 BI 行业关注知乎/专业论坛，SaaS 行业关注 G2/Capterra）。
3. **技术架构**在开发者工具和基础设施行业尤为重要，在消费品行业则可弱化。
4. **合规与安全**在金融、医疗、政府行业是必填维度，应通过动态字段补充。
5. 对比矩阵的**可读性优先于覆盖面**，不是维度越多越好，应聚焦决策者真正关心的差异点。

---

## 质量保障规则

### 分析阶段质量规则

1. **证据追溯**：每个分析结论必须关联 `evidence_refs`，禁止凭空编造。
2. **不完整处理**：当证据只能支撑部分结论时，输出"已确认 + 未确认边界"，不笼统写"无法获取"。
3. **置信度校准**：
   - ≥0.8：多源交叉验证确认
   - 0.6-0.8：单一高质量来源或部分验证
   - 0.4-0.6：推断或间接证据支撑
   - <0.4：证据严重不足
4. **缺失字段处理**：完全无有效信息时，summary 填 `"unknown"`，evidence_gaps 填具体缺失说明。

### 报告生成质量规则

1. **章节完整性**：8 个章节 + 执行摘要 + 参考来源，不得跳章。
2. **字段名一致**：报告中的字段引用必须与 `analysis_schema_plan` 中的 `field_name` 严格一致。
3. **默认语言**：中文输出，除非 `language` 参数明确指定为英文。
4. **可交付性**：输出的 markdown 和 html 应可直接用于汇报或进一步编辑。
5. **风格要求**：正式、简洁、面向产品经理/业务负责人，不使用内部 schema 术语。

### QA 审查通过标准

- 关键结论有可追溯证据
- 不存在完全空白的核心字段
- 对比矩阵覆盖所有计划竞品和字段
- SWOT 条目可执行、可追溯
- 字段覆盖率 ≥ 80%（允许部分证据不足的字段保留并标记）

## 注意事项

1. **证据驱动**：所有分析结论必须基于采集到的真实网页证据，禁止编造数据和事实。
2. **时效性**：关注证据的发布时间，优先使用近 1-2 年的信息，避免使用过时数据。
3. **多源交叉验证**：关键结论（定价、市场份额、核心功能）尽量从 2 个以上独立来源验证。
4. **行业术语**：报告中应使用行业通用术语，避免内部黑话或 schema 字段名直接出现。
5. **保守原则**：当证据不足时，宁可使用"基于公开信息观察"、"需进一步确认"等保守措辞，也不做过度推广性的结论。
6. **跨行业可迁移**：本技能的字段体系和分析框架适用于任何行业，只需调整动态字段和行业适配重点即可。
7. **动态字段不是"可选"**：虽然动态字段由系统自动生成，但一旦纳入分析计划，其重要性等同于核心字段，不得在矩阵或报告中遗漏。
